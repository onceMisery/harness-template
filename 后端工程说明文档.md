## 📌 Harness 后端工程模板说明

本项目采用 **Harness 工程化开发模式**，目标是：

* 将需求 → 实现 → 测试 → 验收流程结构化
* 让 Code Agent 在可控流程下稳定运行
* 提升交付效率与质量

---

# 一、核心概念

## 1. `.harness/` —— 项目唯一事实源

所有工程状态、规划、进度、验收结果都在这里维护：

* 任何开发决策必须基于 `.harness/`
* `feature_list.json` 是唯一可信功能来源
* `run_state.json` 是当前执行状态

---

## 2. 三个核心角色

| 角色        | 职责         |
|-----------|------------|
| Planner   | 把需求转为结构化规划 |
| Generator | 实现功能       |
| Evaluator | 测试与验收      |

---

## 3. 执行流程

```text
start-harness
   ↓
planner
   ↓
generator (按 sprint)
   ↓
evaluator
   ↓
不通过 → 返回 generator
通过 → 下一个 sprint
```

```json
  },

    "planner": ".agents/planner.md",
    "generator": ".agents/generator.md",
    "evaluator": ".agents/evaluator.md"
  }
}
```

---

### 2. `feature_list.json`

**作用：功能唯一来源（最重要）**

```json
[
  {
    "id": "FEAT-001",
    "name": "创建订单",
    "module": "order",
    "type": "backend_api",
    "priority": "P0",
    "status": "planned",
    "passes": false,
    "dependencies": [],
    "inputs": ["user_id", "items", "address_id"],
    "outputs": ["order_id", "status", "amount"],
    "acceptance_criteria": [
      "请求参数校验完整",
      "订单创建成功并持久化",
      "返回统一响应结构",
      "异常场景有明确错误码"
    ],
    "test_scope": ["unit", "integration", "contract"],
    "owner": "generator",
    "last_eval_score": 0,
    "notes": ""
  }
]
```

✅ 编写要求：

* 每个功能必须有 ID
* 必须写 acceptance criteria
* 不允许模糊描述

## `.agents/`

定义 AI 行为

### planner.md

* 只做规划
* 不写代码
```text
# Planner Agent

## Role
你负责把需求转化为项目可执行资产，不编写业务代码。

## Must Read
- .skills/harness-core/SKILL.md
- .skills/harness-planning/SKILL.md
- .harness/project.json
- existing .harness/* files

## Responsibilities
- 生成或更新 spec.md
- 生成或更新 feature_list.json
- 生成或更新 sprint_plan.json
- 补充 domain_model.md / api_contracts 如需要
- 标记 assumptions / risks / out-of-scope

## Must Not
- 不得实现业务代码
- 不得跳过 acceptance criteria
- 不得生成未授权功能

## Completion Signal
输出：PLAN_READY
并确保所有规划文件通过 schema 校验
```

### generator.md

* 只实现当前 sprint
* 不扩需求
```text
# Generator Agent

## Role
你负责实现当前 sprint 已批准范围内的功能。

## Must Read
- .skills/harness-core/SKILL.md
- .skills/harness-generation/SKILL.md
- .harness/feature_list.json
- .harness/sprint_plan.json
- .harness/run_state.json
- current sprint contract

## Responsibilities
- 仅实现当前 sprint feature
- 仅修改 allowed changes 范围内文件
- 为变更补充对应测试
- 更新 handoff / progress / feature 状态

## Must Not
- 不得跨 sprint 扩 scope
- 不得修改 forbidden changes
- 不得自评通过

## Completion Signal
输出：GENERATED
并生成 sprint handoff
```

### evaluator.md

* 只测试
* 不修改代码

```text
# Evaluator Agent

## Role
你负责验证成果是否达到放行标准。你的职责是发现问题，而不是赞美结果。

## Must Read
- .skills/harness-core/SKILL.md
- .skills/harness-evaluation/SKILL.md
- current sprint contract
- sprint handoff
- feature_list.json

## Responsibilities
- 执行测试
- 根据 acceptance criteria 验收
- 输出 blocker / major / minor issue
- 生成 evaluator report
- 更新 feature passes / score

## Must Not
- 不得实现修复
- 不得跳过失败项
- 不得在存在 blocker 时判定通过

## Completion Signal
输出：PASSED 或 REWORK_REQUIRED
```

# 三、开发流程（必须遵守）

## Step 1：写 spec + feature

只允许 Planner 修改：

* spec.md
* feature_list.json
* sprint_plan.json

---

## Step 2：执行 sprint

Generator：

* 实现功能
* 写测试
* 更新 handoff

---

## Step 3：执行评估

Evaluator：

* 跑测试
* 输出评分
* 判定是否通过

---

## Step 4：循环修复

```text
不通过 → generator 修复 → evaluator
```

---

## Step 5：完成条件

```text
feature_list.json 中全部 passes=true
```

---

# 四、强制规则（重要）

必须遵守：

1. 禁止跳过 planner
2. 禁止 generator 自评
3. 禁止修改未授权模块
4. 未通过 evaluator 不允许进入下一 sprint
5. 所有状态必须写入 `.harness/`

---

# 五、最佳实践

## 1. Feature 要小

不要一个 feature 做太多事

## 2. Sprint 控制在 2~4 个 feature

## 3. 测试必须覆盖

* unit
* integration
* contract

## 4. 先保证主链路

再做边界优化
